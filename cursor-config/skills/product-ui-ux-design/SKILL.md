# Product UI/UX Design（全プロジェクト共通）

## いつ使うか

| 状況 | 主に読むもの |
|------|----------------|
| **アプリ・ダッシュボード・社内ツール・データ画面**の体験・視覚の再設計 | **本 SKILL** |
| **LP・コーポ・サービス紹介ページ**（`docs/site` 運用） | `site-build-orchestrator` + `b2b-site-strategy-and-ia` + `site-design-planner` |
| 実装レベルの Tailwind 統制 | `tailwind-product-ui-conventions` |
| マーケ向け motion / セクション | `shadcn-motion-ui-brief` |
| 実装後の品質確認 | `review-frontend` / `review-app-product` |

**本 SKILL は「プロダクト体験の設計プロセス」**、サイト専用オーケストレーションは **上表のとおり分岐**する。

## 外部知見の要点（AI 補助デザインで共通する結論）

- **分担**: AI はバリエーション生成・トークン適合・コントラスト確認・下書きに強い。**戦略・ブランドの機微・IA・最終責任は人間**（並列hq.com / 業界記事の合致: 設計システムに「接地」させないと出力は似通う）。
- **反復**: 初回プロンプト一発完結を期待しない。**2〜3 サイクル**で意図に寄せるのが通常。
- **システム接地**: トークン・コンポーネント規約・禁止パターンを **文書化してから** 実装へ渡すと「もっともらしいがズレ」が減る。

詳細ヒューリスティクスは [reference-heuristics.md](reference-heuristics.md)。

## 設計フロー（この順で出力）

### 0. ユーザーファーストの固定（必須）

- 先に **`Read`**: [user-first-product-design](../user-first-product-design/SKILL.md) のチェックリストを埋め、本 SKILL の §1〜8 と矛盾がないようにする。
- **圧倒的に使いやすいか**（学習・迷い・取り戻し・待機）を §1 の成功指標に必ず接続する。

### 1. 文脈とジョブ

- **誰が**（主ペルソナ 1〜2）、**どんな状況で**、**何を達成したいか**（JTBD 1 行）。
- **成功指標**（例: タスク完了時間、エラー率、理解度、NPS は任意）。
- **非目標**（やりたくないこと・スコープ外）。

### 2. 情報の優先順位（1画面あたり）

- **Primary action**（1 つに絞れるか）。
- **Secondary / tertiary** の並べ方（視線＝重要度と一致しているか）。
- **Above the fold** に載せる情報の上限（詰め込みすぎない）。

### 3. フローとナビゲーション

- **Happy path** のステップ数（3 タップルールは目安、硬直させない）。
- **迷子時の脱出**（パンくず、戻る、キャンセル、ホーム）。
- **URL / 画面名** とユーザー心理モデルの一致。

### 4. 状態設計（必須）

各主要画面について表形式で定義する:

| 状態 | ユーザーに見せるもの | システムの振る舞い |
|------|----------------------|-------------------|
| 初期 / 空 | | |
| 読込中 | | |
| 部分データ | | |
| エラー（回復可能/不可） | | |
| 成功 | | |

**空状態**に説明だけで終わらせず、**次のアクション**を 1 つ置く。

### 5. ヒューリスティック・クイックパス

[reference-heuristics.md](reference-heuristics.md) の Nielsen を、**今回の画面で危うい 3 点だけ**列挙し、対策を 1 行ずつ。

### 6. 視覚方針（「生成 UI 臭」を避ける）

以下を **実際に決める**（デフォルト禁止の羅列ではなく、案件で選択）:

- **タイポ**: 和文を主とするなら、読みやすい日本語ウェブフォントまたはシステム UI フォント＋英字のペアリング方針。
- **色**: プライマリ 1、アクセント 1、**意味のある色**（成功/警告/危険）のルール。装飾グラデで階層を誤魔化さない。
- **密度**: 「情報ダッシュ」と「余白のある説明」のどちらを基準にするか。
- **角丸・影**: 1 段階か 2 段階か。**コンポーネントごとに微妙に違う角丸**を禁止。

**ストレステスト**: 見出しが **2 行に折り返す**、数値が **桁増えする**、ラベルが **長い日本語**でもレイアウトが破綻しないかを文章で宣言する。

### 7. アクセシビリティ（実装前の約束）

- キーボードで主要フローを完走できるか。
- コントラスト方針（トークンで担保するか）。
- `prefers-reduced-motion` の扱い（アニメ前提の UI にしない）。

### 8. ハンドオフ（実装者向け）

- **コンポーネント粒度**の目安（ページに書き散らかさない単位）。
- **既存スタック**（例: shadcn / Tailwind）が決まっているなら、**どの primitive を使うか**の地図。
- **受け入れ条件**を Given–When–Then で 3 本以上。

実装フェーズに入ったら **`tailwind-product-ui-conventions`** と **`test-driven-development`** / **`review-frontend`** を読む。
